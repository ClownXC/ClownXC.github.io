1. 概述

1.1. SQL 执行时间长的原因

  • 查询语句写的烂
  • 索引失效
  • 关联查询有太多 Join
  • 服务器调优及各个参数设置，如缓冲、线程数等

1.2. SQL 执行顺序

  • FROM

    sql 首先执行 from 将数据从硬盘加载到数据缓冲区中，以便对这些数据进行操作。如果有多个表则求所有表的笛卡尔积。
    [ table1的记录条数 * table2的记录条数 * tableN的记录条数）这时第一个虚拟表产生VT1]

  • ON

    其次执行on，从 VT1 中取出匹配 on 条件的行, 产生 VT2。

  • JOIN

    left join，right join 外部连接就是在这个时候执行的，在 VT2 的基础上添加符合条件的外部的行，产生 VT3。

  • WHERE

    执行 where 过滤，产生 VT4
    由于这个时候 select 没有执行，所以 select 中的别名不可以用于 where

  • GROUP BY

    在 VT4 的基础上执行 group by 分组数据，产生 VT5。

  • HAVING

    在 VT5 的基础上执行 having 过滤，产生 VT6。

  • SELECT

    查出我们要的字段，产生 VT7

  • DISTINCT

  • ORDER BY

    排序 产生 VT8

  • LIMIT

    限制返回行数

2. SQL 性能分析

3. Explain

Explain 命令，可以用显示 MySQL 如何使用索引来处理 select 语句以及连接表

3.1. 字段

3.1.1. id

  • 概述

    查询序号，表示查询中 select 语句执行顺序

  • 说明

    • id 相同时，执行顺序由上至下
    • 子查询 id 的序号会递增，id 值越大优先级越高，越先被执行
    • id 相同，可以视为一组，从上往下顺序执行

3.2.2. select_type

  • 概述

    查询类型，主要是用于区别普通查询、联合查询、字查询等复杂查询

  • 说明

    • SIMPLE 表示简单的 select，没有 union 和子查询
    • PRIMARY 子查询的语句中，最外面的 select 查询就是 primary
    • UNION：union 语句的第二个或者说是后面那一个
    • DERIVED在 from 列表中的子查询被标记为 DERIVED[衍生]，MySQL会递归执行这些子查询，并把结果放在临时表中。

3.2.3. table

显示这一步所访问数据库中表名称[显示这一行的数据是关于哪张表的]，有时不是真实的表名字，可能是简称

3.2.4. type

表示 MySQL 在表中找到所需行的方式，又称**”访问类型”**

常用的类型有:ALL, index, range, ref, eq_ref, const, system [从左到右，性能从差到好]

  • ALL

    Full Table Scan， MySQL 将遍历全表以找到匹配的行

  • index

    Full Index Scan，遍历所有的索引树，比 ALL 要快的多，因为索引文件要比数据文件小的多

  • range

    查找某个索引的部分索引，一般在 where子句中使用 < 、>、in、between 等关键词。只检索给定范围的行，属于范围查找

  • ref

    查找非唯一性索引**，返回匹配某一条件的多条数据**。属于精确查找、数据返回可能是多条

  • eq_ref

    查找唯一性索引，返回的数据至多一条。属于精确查找

  • const

    查找主键索引，返回的数据至多一条（0或者1条）。属于精确查找（id为主键）

  • system

    查找主键索引，返回的数据至多一条（0或者1条）。属于精确查找

3.2.5. possible_keys

显示可能应用在这张表中的索引，一个或多个

查询涉及到的字段上若存在索引，则该索引将被列出，但不一定被查询实际使用。

3.2.6. key

显示 MySQL 实际决定使用的索引

如果没有选择索引，键是 NULL。

要想强制 MySQL 使用或忽视 possible_keys 列中的索引，在查询中使用 FORCE INDEX、USE INDEX 或者IGNORE INDEX

3.2.7. key_len

表示索引中使用的字节数，可通过该列计算查询中使用的索引的长度（key_len显示的值为索引字段的最大可能长度，并非实际使用长度，即 key_len 是根据表定义计算而得，不是通过表内检索出的）

key_len公式

  • varchar(10) 变长字段并且允许null

    key_len = 10 * (character set: utf8=3, gbk=3, latin=1) + 1 [null] + 2 [变长字段]

  • varchar(10) 变长字段不允许null

    key_len = 10 * (character set: utf8=3, gbk=3, latin=1) + 2 [变长字段]

  • char(10) 固定字段并且允许null

    key_len = 10 * (character set: utf8=3, gbk=3, latin=1) + 1 [null]

  • char(10) 固定字段不允许null

    key_len = 10 * (character set: utf8=3, gbk=3, latin=1)

3.2.8. ref

3.2.9. rows

MySQL 估计为了找到所需的行而要读取的行，rows 值很重要，它决定采用哪个索引以及是否放弃索引改为全表。它是一个平均数，用来估算查找需要行而必须读取的平均值。

3.2.10. 6.extra

  • Using filesort

    说明 MySQL会对数据使用一个外部的索引排序，而不是按照表内的索引顺序进行读取

  • Using temporary

  • Using index

    出现这个说明 MySQL 使用了覆盖索引，避免访问了表的数据行，效率不错！通俗的说也就是查询的列不需要回表，在索引树上就能拿到结果

  • Using where

    说明服务器在存储引擎收到行后将进行过滤。有些where中的条件会有属于索引的列，当它读取使用索引的时候，就会被过滤，所以会出现有些where语句并没有在extra列中出现using where这么一个说明。

  • Using join buffer
