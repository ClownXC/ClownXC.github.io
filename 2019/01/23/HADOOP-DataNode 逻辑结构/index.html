Datanode 逻辑结构
发表于2019-01-23|更新于2021-05-24|Hadoop
|字数总计:837|阅读时长:2分钟|阅读量:

Datanode 逻辑结构图，Datanode 从逻辑上可以切分为如下几个模块。

1. 数据层

Datanode 的所有服务都是建立在数据块存储功能基础上的，将 Datanode 中负责在本地磁盘存储数据块的部分抽象为数据层。 数据层包括两个主要的模块。

1.1. DataStorage 数据存储

DataStorage 负责管理与组织 Datanode 的磁盘存储空间，同时也负责管理存储空间的生命周期（包括升级、回滚、 提交等操作）。在 HDFS Federation 架构中，一个 Datanode 可以保存多个块池的数据块，HDFS 通过 BlockPoolSliceStorage 管理 Datanode 上单个块池的存储空间。 DataStorage 持有所有 BlockPoolSliceStorage 对象的引用，并通过这些引用管理 Datanode 上的所有块池。

1.2. FsDataset 文件系统数据集

FsDataset 抽象了 Datanode 管理数据块的所有操作，创建数据块文件、维护数据块文件和校验和文件的对应关系等。Datanode 都可以配置若干个不同类型的存储目录来保存数据块，通过配置项 HDFS 通过 FSVolumeImpl类来管理 Datanode上单个存储目录保存的所有数据块，同时定义了 FSVolumeList 类来维护 Datanode上所有FSVolumeImpl 对象的引用。FsDataset会通过 FSVolumeList 类提供的功能管理Datanode上所有存储目录保存的数据块。

2. 逻辑层

在数据层基础上，Datanode 执行若干 HDFS 逻辑，例如向 Namenode 汇报数据块存储状态、发送心跳、扫描损坏的数据块等，将 Datanode 上负责执行 HDFS 逻辑的部分抽象为逻辑层。 逻辑层包括三个主要的模块。

2.1. BlockPoolManager

BlockPoolManager 管理所有块池的接口类。联邦机制下每个 HDFS 集群都可以配置多个命名空间，每个命名空间在 Datanode 上都有一个与之对应的块池。一个 BlockPoolManager 对象会持有多个 BPOfferService 对象的实例，每个 BPOfferService 对象都管理这个 Datanode 的一个块池。每个命名空间定义两个 Namenode，一个作为 Active Namenode，一个作为 Standby Namenode。所以每个 BPOfferService 对象又都会持有两个 BPServiceActor 对象，一个 BPServiceActor 对象对应于命名空间中的一个 Namenode，该对象负责向这个Namenode 发送心跳、块汇报、缓存汇报以及增量块汇报，并执行Namenode 返回的名字节点指令（DatanodeCommand)

2.2. DataBlockScanner

一个独立的线程，周期性地扫描每个数据块并检查数据块的校验和是否正常

2.3. DirectoryScanner

一个独立的线程，定时发起对磁盘数据块的扫描，对比内存中元数据与实际磁盘存储数据块的差异，然后更新内存中的元数据，使之与磁盘保存的数据块信息一致

3. 服务层

Datanode 还包括一个对外服务层，服务层中的模块用于支持其他节点与 Datanode通信，以及访问 Datanode 状态等功能。服务层包括三个模块。

3.1. Http Server

对外提供 HTTP 服务，可用于展示Datanode内部状态

3.2. IPCServer

RPC 服务端，响应来自于 Client，Namenode以及其他 Datanode的RPC请求。

3.3. DataXceiverServer

数据传输服务端，响应来自于Client 以及其他 Datanode的流式接口请求。
