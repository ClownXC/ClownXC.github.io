Scilab 云化改造(4)_Xcos工具箱功能包挂载
发表于2018-03-30|更新于2021-01-19|Scilab
|字数总计:4k|阅读时长:19分钟|阅读量:

为实现 Scilab 扩展功能的增加和挂载，将 C/C++/Fortran 语言实现的功能添加到 Scilab 平台，并在 Scilab 平台中对该类函数实现的功能进行调用。Scilab 提供了工具箱的模式对此类功能进行管理。

工具箱概述

工具箱结构

scilab 工具箱有专门的层次结构和概念，包括以下几部分：

  1. script

    script 是一个扩展名为.sce的脚本文件，用于实现Scilab批处理代码。

  2. macro

    macro是用Scilab代码编写的Scilab函数，是一些具体功能的实现，其文件扩展名为.sci。

  3. primitive

    primitive是Scilab可调用的外部函数，该类函数的原型可以是C/C ++/Fortran语言编写的。该类型的函数需要通过sci_gateway的方式实现调用。

  4. sci_gateway

    sci_gateway是调用外部函数的模式方法，其方法使用C或C ++编写，遵循一定的原则，调用外部的功能实现函数，使其可以从Scilab中直接调用。一旦在builder脚本中声明了网关函数，构建并加载了其所对应的功能函数，就可以直接从Scilab中使用Scilab的方法使用这些外部实现函数。

  5. builder

    builder文件是用于构建工具箱的Scilab脚本，即对工具箱源码进行编译并生成对应的库文件。sci文件生成对应的bin文件，C/C++/Fortran文件生成对应的dll/so等动态库文件。以及符合Scilab加载的loader脚本文件、卸载的unloader脚本，以及清理生成文件的cleaner脚本。

  6. loader

    loader是Scilab脚本，它在Scilab中加载工具箱组件（或整个工具箱）。加载程序名称包含加载程序* .sce（示例：loader.sce，loadhelp.sce，…）。该文件由对应的builder文件生成。

  7. Toolbox start & quit

    每个通过ATOMS安装的工具箱都在Scilab启动时自动加载启动脚本，在Scilab退出时自动执行工具箱卸载脚本。

Scilab 的工具箱构建时一般遵循以下结构：

目录 内容 备注
macros macros(.sci)，builder(buildmacros.sce) 必需
src 源代码文件（.c，.cpp，.f，…）和builder脚本（builder*.sce）和加载脚本（loader * .sce，由builder脚本产生） 必需
sci_gateway 网关（.c或.cpp）以及网关builder脚本（.sce）的源代码文件 可选，如果没有使用其他语言实现的功能时，可忽略(如celestlab)
jar java包和帮助文件（.jar） 不需构建，由builder运行时自动生成帮助文档的jar
help XML帮助文件（.xml） 可选，但建议添加必要的帮助说明
etc 启动脚本（.start）和退出（.quit）脚本 必需
tests 模块测试脚本（.tst） 可选
demos 脚本实例 可选，但建议添加必要的示例
includes 要与模块一起发布的头文件（.h） 可选，如果想让别的程序调用本工具箱的功能，可以通过该方法进行分享

工具箱加载方式

工具箱加载时需要两类脚本：一个是编译脚本(builder.sce)，用于编译工具箱; 另一个是加载脚本(loader.sce)，在Scilab 中加载该工具箱时运行。两者都是 Scilab 的批处理脚本文件（.sce文件）。

编译工具箱时，运行 builder.sce 文件，在编译过程中，该脚本将去各个源码目录下执行各自的builder脚本，完成各个源码的编译工作。编译完成后，可以使用加载脚本（loader.sce），在Scilab中加载模块。加载过程中，加载脚本会去工具箱/etc/目录下执行工具箱的启动脚本.start。

功能包概述

功能包结构

Scilab的软件架构是模块化结构，即每一类功能都类似于插件。现对源码中的层次架构进行说明 。SiROS

在最上面的文件层次中，如图所示，层级目录中有10个文件夹。每个文件夹都有具体的含义，因为本文档针对的是对功能包的挂载，所以在本报告中需要关注的是modules文件夹，其中共有77个子文件夹，每一个子文件夹代表了其实现的一种功能，即每个子文件夹都是基本上代表了一类功能包。

现对功能组件包进行层级说明，以一般功能包为例。

Scilab 功能包构建层次说明

目录 内容 备注
demos 模块的演示，显示模块的功能
etc 模块配置（初始化和配置）
example 示例（readme.txt）
help 功能包的帮助（SiROS帮助说明）
includes 可以在外部使用的头文件（其他功能包或第三方程序）
locales 本地化文件
macros 模块的宏脚本文件
sci_getway 链接外部程序到SiROS中 子目录应为(sci_getway/c/ sci_getway/fortran/…)
src 功能包的核心源码 子目录应为(src/c/ src/fortran/…包括一些本地.h头文件)
tests/unit_tests 单一测试文件
tests/nonreg_tests 非递归测试文件
tests/benchmarks 基准测试文件

其中在一个功能包中，所必须的文件包括：

  1. changelog.txt
  2. licence.txt
  3. Makefile.am
  4. Makefile.in (generated from Makefile.am by automake)
  5. xxxx.vcproj
  6. version.xml
  7. readme.txt
  8. etc/.start
  9. etc/.quit
  10. includes/gw_.h
  11. sci_gateway/xxx/gw_.c
  12. sci_gateway/_gateway.xml

可选项为：

locales/en_US/.po

需要遵守的原则为：

如果它们是模块的native函数，则应在src / /生成函数的.h头文件。如果可以从其他模块使用该函数，则应将其头文件放入includes /。

应该尽量避免使用extern字段。可以使用头文件（.h）以实现相同的功能。

当创建可以由fortran调用的C函数时，需要在函数名称前添加宏C2F，并需要引入 machine.h 头文件。

功能包加载方式

功能包的加载方式，由于其是与源码一起编译生成相应的 dll/so 文件，所以其加载方式是在 Scilab 启动时，直接自动加载其对应的 dll/so 文件或者在需要时由 Scilab 决定其加载与否。所以我们需要关注的是其编译的过程而非加载过程。

下一小节将以 xcos 实现编译的 Makefile 文件为例，进行说明。

功能包示例

src/modules/xcos/Makefile.am

另外在添加或移除功能包时，亦需要修改

src/modules/Makefile.am

src/Makefile.am

src/configure.ac

这三项文件，然后在源码所在目录下的终端执行以下命令：

[user@localhost src] autoconf

此命令根据configure.ac生成configure文件

[user@localhost src] automake

此命令根据Makefile.am生成Makefile.in文件

[user@localhost src] ./configure && make && make install

此命令对源码进行重新编译。

功能包与工具箱对比

工具箱加载时，运行主构建器文件（builder.sce），由主构建器文件调用个文件夹中的构建器文件（builder_**.sce），分别生成各加载文件，等待各加载文件构建成功后，再加载主加载文件，将工具箱加载到Scilab平台。

Scilab工具箱的层次结构十分严谨，src文件夹内的函数需要与sci_gateway文件夹内的网关函数紧密对应；各文件夹内的builder文件（builder*.sce）需要与各自文件夹内的文件对应，尤其是src中的c文件和sci_gateway中的c文件。

实际上，功能包和工具箱的结构类似，实现的本质上是一样的。这是都是因为基于Scilab是一个模块化架构的软件。也就是说，功能包和工具箱是可以相互转化的。功能包可以按照工具箱的加载方式进行加载。工具箱也可以根据功能包的方式实现与Scilab内联，即成为Scilab基本的一部分。
